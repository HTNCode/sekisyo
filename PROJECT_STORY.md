# Sekisyo CLI — 理解してから、レビューへ。

> 私たちは、AIが書いたコードを拒まない。説明のないコードを拒む。

- Sekisyo
  CLIは、AI生成コードを禁止するためのツールではありません。コードをレビューへ届ける前に、作業者自身が変更内容と設計判断を説明できる状態へ導く、Gitの`pre-push`フック型CLIです。

## インスピレーション

- 日々の開発現場でAI駆動開発が進むにつれて、開発者がコードを書く速度は大きく向上しました。一方で、人間がコードを理解し、判断し、レビューできる速度は、同じようには増えていません。
- 実際の開発では、AIが生成したコードの中身や設計判断を作業者が十分に理解しないまま、レビューへ提出してしまう場面が増えていると感じていました。その状態では、レビュアーがコードの正しさだけでなく、「なぜこの変更をしたのか」「どの境界条件を考慮したのか」「既存機能へどのような影響があるのか」まで、差分から推測しなければなりません。
- AIによって実装は速くなっても、レビューに使える時間と注意力は増えません。その結果、人間のレビュー帯域が新しいボトルネックになります。
- これは短期的なレビュー工数だけの問題ではありません。作業者がコードを読み、設計判断を言葉にする機会が減れば、中長期的にはチームの知識蓄積や人材育成にも影響します。「理解する責任」が作業者からレビュアーへ移り続ける状態は、AIで開発を効率化したはずの組織に、新しい負債を生みます。
- 既存のAIレビューサービスには、PRへ自動で指摘を付けたり、レビュアーの作業を支援したりするものが多くあります。しかし私が求めていたのは、もっと上流で、コードを書いた側の理解と学習を促す教育的な仕組みでした。
- 同時に、AI生成コードを検出して禁止する方法にも違和感がありました。必要なのはAIを使わない開発へ戻ることではなく、AIを使う前提を崩さずに、その使い方をより良くすることです。
- そこで、既存の開発フローを大きく変えず、レビューへ進む直前に一度だけ立ち止まる「関所」を置くという発想から、Sekisyo
  CLIを形にしました。

## 機能

- Sekisyo
  CLIはGitの`pre-push`フックとして動作します。作業者が普段どおり`git push`すると、コードをリモートへ送る前に次の処理を行います。

1. **差分を理解し、読む場所を絞る**
   Codexがリポジトリ全体の文脈を踏まえ、変更箇所と影響範囲を分析します。同時に、差分を「機械的変更・定型・必読」へ分類した注意力マップと、push前に作業者が確認すべき指摘を作ります。

2. **一次セルフレビューを行う**
   Codexの指摘ごとに、作業者は「修正するため中断」するか、「意図的な変更」として受け入れる理由を説明します。理由は後でPR本文にも残ります。

3. **作業者へ問いかける**
   GPT-5.6が、変更内容を本当に理解していなければ答えにくい質問を生成します。

4. **曖昧な説明を掘り下げる**
   「たぶん問題ありません」のような回答には、関連箇所や設計判断を尋ねる追加質問を返します。

5. **説明できる状態でpushする**
   回答が具体的になったら、Q&Aと要約をHEADのSHA、base、push先ref、質問ポリシー、モデルに紐づけて一時保存し、pushを続行します。

6. **レビューへ判断材料を渡す**
   `sekisyo pr`は、注意力マップ、設計判断、認識済みリスク、検証内容、未解決事項、一次セルフレビューの理由、合格したQ&AをPR本文の管理ブロックへ作成または更新します。書き出しに成功したローカルの通行手形は削除します。

質問は、単にコードを要約させるものではありません。Sekisyo
CLIでは「理解していなければ答えにくい問い」を、次のタクソノミーとして定義し利用します。

| 質問の型           | 確認すること                                       |
| ------------------ | -------------------------------------------------- |
| 境界条件           | 空の入力や最大値などで、処理がどう振る舞うか       |
| 非変更側への影響   | 変更していない呼び出し元や既存機能へ何が波及するか |
| 選ばなかった代替案 | なぜ既存実装や別の設計を採用しなかったのか         |
| 失敗時の挙動       | タイムアウトや途中失敗時に、どの状態が残るか       |

```text
$ git push

Q. この変更でinvalidateを呼ばなくなった旧経路のキャッシュは、誰が無効化しますか？

> たぶん問題ありません

その回答では影響範囲を確認できません。関連する呼び出し元と無効化のタイミングを説明してください。
```

- 試問への回答を、すべて人間だけで書く必要はありません。AIと相談しながら答えても構いません。
- 中身のある説明をAIに作らせるためには、作業者が差分を読み、必要な情報を与え、生成された説明を検証する必要があります。その過程でコードへの理解が深まるなら、それもSekisyo
  CLIが目指す学習の形です。
- 判定するのは「人間が書いたように見えるか」ではなく、変更に対して具体的で検証可能な説明になっているかどうかです。

## 開発方法

### Gitの自然な境界に置く

- レビュー依頼を出した後ではなく、コードをリモートへ送る直前に介入するため、Gitの`pre-push`フックを採用しました。
- これにより、GitHub
  Actionsの実行を待つ必要がなく、関所そのものはGitHub、GitLab、自社Gitなど、特定のホスティングサービスに依存しません。`sekisyo pr`によるPR本文の更新だけは、現在のMVPではGitHub
  CLIを利用します。
- IDEやAIエージェントからpushした場合もフックは発火します。ただし対話端末を利用できないpushは安全側で停止し、先に端末で`sekisyo ask`を完了するよう案内します。

### Codexでリポジトリ全体を分析する

- Codex
  CLIをheadlessで実行し、変更された行だけではなく、呼び出し元や関連ファイルを含むリポジトリ全体の文脈から差分を分析します。
- 良い質問を作るには、変更コードの要約だけでは不十分です。「変更していない箇所へ何が波及するか」を見つけられることが重要でした。

### GPT-5.6で質問生成と回答判定を行う

- Codexの分析結果をもとに、GPT-5.6が質問タクソノミーに沿った問いを生成します。さらに、作業者の回答に対象箇所、失敗条件、設計判断などの具体性が含まれているかを評価し、必要に応じて追加質問を返します。
- Sekisyo
  CLIの中心にあるのは、「答えを出すAI」ではなく、**理解を引き出す良い問いを出すAI**です。

### サーバーやデータベースを持たない

- 通過状態とQ&Aは、HEADのSHAなどの通過条件に紐づけて`.git/sekisyo/`へ一時保存します。worktreeの外側に置くため、誤ってコミットへ含まれることもありません。
- 記録を恒久保存するために`git notes`を使う案も検討しましたが、追加のpush設定、rebaseやsquash
  mergeとの相性、GitHub UIから見えにくいことなどを考え、採用しませんでした。

| 要素           | 採用技術             | 役割                                  |
| -------------- | -------------------- | ------------------------------------- |
| CLI            | TypeScript + Bun     | 高速に起動し、配布しやすいCLI         |
| 実行タイミング | Git `pre-push`フック | レビューへ届く直前の関所              |
| 差分分析       | Codex CLI            | リポジトリ全体の文脈を含む分析        |
| 質問と判定     | GPT-5.6 API          | 質問生成、回答判定、追加質問、Q&A要約 |
| 一時記録       | `.git/sekisyo/`      | サーバー不要のローカル状態管理        |
| PR連携         | GitHub CLI           | 注意力マップと説明記録をPR本文へ反映  |

## 直面した課題

### 良い質問と、ただの要約質問を分けること

- モデルへ単純に質問生成を依頼すると、「この関数は何をしていますか」といった、コードを読み上げるだけでも答えられる質問が生まれがちです。しかし、それでは変更のリスクや設計判断を理解しているか確認できません。
- そこで、境界条件、影響範囲、代替案、失敗時の挙動へ問いを分解し、質問タクソノミーとして明示しました。プロンプト内部の工夫だけにせず、「何を問うべきか」自体をプロダクトの仕様として設計しました。

### 教育と監視の境界を設計すること

- 関所という仕組みは、設計を誤ると作業者を監視し、AI利用を取り締まるツールになってしまいます。
- Sekisyo
  CLIは、AIを使ったかどうかを検出しません。試問への回答にAIを使うことも許容します。また、`git push --no-verify`でバイパスできることも許容します。
- 目的は人を取り締まることではなく、作業者自身が説明責任を果たし、理解を深めるための道具を提供することです。
- あくまで組織ごとの成長度に合わせて柔軟に選択できる運用の自由度を残しています。

### 立ち止まる価値と開発速度を両立すること

- 質問が多すぎれば、Sekisyo
  CLI自体が新しいボトルネックになります。逆に、簡単すぎれば関所として意味がありません。
- すべての変更を同じ深さで問うのではなく、リスクの高い箇所や人間が判断すべき箇所へ質問を集中させる必要がありました。レビューを置き換えるのではなく、レビュー前に理解と注意の向け先を整えることを重視しました。

### 一時記録と恒久記録を分けること

- 通過記録を永続的に蓄積すると、rebaseやsquash
  mergeのたびに管理コストが発生します。一方で、設計判断はチームの知識として残す価値があります。
- そこで、ローカルの通行手形は使い捨てにし、長く残す価値のある合格Q&A、設計判断、リスク、検証内容、未解決事項、注意力マップをPR本文へ移すというライフサイクルを設計しました。

## 誇りに思う成果

- 最も誇りに思うのは、差分分析、口頭試問、回答の深掘り、pushの続行という一連の体験を、作業者が普段使っている`git push`の中へ収められたことです。新しい管理画面や複雑なサーバーを導入せず、既存の開発フローを崩さない形で「理解するための関所」を作れました。
- また、AI生成コードを検出したり禁止したりするのではなく、**説明の具体性だけを問う**という立場を、プロダクト全体で一貫させられたことも成果です。AIと一緒に回答することさえ許容しながら、作業者の理解と説明責任を促す設計にできました。
- さらに、質問の品質をモデル任せにせず、「境界条件」「非変更側への影響」「選ばなかった代替案」「失敗時の挙動」という公開可能なタクソノミーへ落とし込めました。良い問いを再現可能なプロダクト仕様に変えられたことは、Sekisyo
  CLI独自の価値だと考えています。
- 作業者には学びを、レビュアーには判断しやすい情報を、チームには設計判断の記録を届ける。この三者の価値を、ひとつの小さなCLIにまとめられたことを誇りに思います。

## 学んだこと

- 最も大きな学びは、AI駆動開発におけるボトルネックが、コード生成から**理解の受け渡し**へ移りつつあるということです。
- AIがコードを速く書けるほど、「なぜこの変更をしたのか」「何が起きたら壊れるのか」を説明する情報の価値が高まります。レビュー負担や組織的な教育における負債を減らしていくためには、レビューを速く的確にできるようにするだけでなく、レビュアーへ届く情報の質を上流で高める必要があります。
- また、AIは答えを生成するだけでなく、人間の理解を深める問いを作るためにも使えると学びました。質問の品質をプロダクトの中心に据えることで、AIを作業の代替だけではなく、学習を支える相棒として使えます。
- そして、AIの使用を禁止しなくても説明責任は高められます。AIと一緒に答えることを認めたうえで、回答の出自ではなく具体性を評価する。この割り切りによって、AI活用と人材育成を対立させずに済みました。

## Sekisyo CLIの現在地と今後の展望

- MVPでは、`pre-push`フック、Codexによる差分分析、一次セルフレビュー、GPT-5.6による口頭試問と回答判定、注意力マップ、ローカル一時記録、PR本文への書き戻し、`.sekisyo.yml`による質問タクソノミーのカスタマイズ、PR本文の記録を確認する検証CIまで実装しました。
- 将来的には、Sekisyo
  CLIを単なるpush前のチェックツールではなく、チームごとの「何を理解し、何を説明できるべきか」を育てる仕組みにしたいと考えています。
- 引き続き目指すのは、AI利用をあくまで推進し、AIが書いたコードを作業者が理解し、レビュアーが判断しやすく、チームの知識として残していけるようにすることです。
